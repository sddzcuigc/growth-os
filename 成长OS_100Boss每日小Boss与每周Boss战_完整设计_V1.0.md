# 成长OS：100能力Boss、每日小Boss与每周Boss战完整设计

> 版本：V1.0  
> 日期：2026-07  
> 用途：产品设计、Codex开发、关卡内容制作与家庭试运行

## 0. 产品定位





要做的**不是儿童版 Todo List，也不只是“AI每天推荐一个任务”**。

你真正想做的是一套：

# AI儿童未来成长操作系统

它先回答五个连续问题：

1. **未来10—20年，一个人最需要哪些底层能力？**
2. **这个孩子现在是谁：有什么兴趣、优势、潜力、问题和环境限制？**
3. **未来能力库与孩子画像之间，应该怎样匹配？**
4. **怎样把长期成长蓝图逐层拆成年、月、周、日计划？**
5. **怎样通过任务执行、作品、经验、宝石和奖励，让孩子长期坚持，并持续修正画像？**

可以概括为：

> **未来能力地图 × 动态天赋画像 × 科学成长模型 × 分层规划系统 × 游戏化执行系统 × 证据反馈闭环**

前面我把“快速开始一个小任务”放得太重了。那只是最底层的**执行模块**，不是你的产品灵魂。

你的产品灵魂其实是：

> **AI先研究未来需要什么，再理解这个孩子是谁，然后为这个孩子规划一条通向未来的个性化成长路线。**

# 一、你这套系统完整的逻辑应该是什么

## 第一层：未来能力地图

系统首先建立一个“未来10—20年核心能力库”。

这不能简单理解为：

- 编程；
- 英语；
- 画画；
- 写作；
- 数学。

这些是**知识、工具或学习载体**，不完全等于未来能力。

未来能力应该是更底层、更可迁移的东西，例如：

- 定义问题；
- 独立判断；
- 与AI协作；
- 识别错误信息；
- 把复杂任务拆解；
- 在不确定条件下决策；
- 从失败中调整策略；
- 表达自己的真实意图；
- 组织资源完成项目；
- 与不同的人合作；
- 长期管理身体和注意力；
- 创造别人愿意使用的成果。

世界经济论坛预计，到2030年，劳动者约39%的现有技能组合可能发生变化；增长最快的不只有AI、大数据、网络安全和技术素养，也包括创造性思维、韧性、灵活性、好奇心和终身学习。

OECD的未来教育框架也没有把未来能力仅仅定义为职业技能，而是强调：

- 学生自主性；
- 幸福与身心状态；
- 知识；
- 技能；
- 态度；
- 价值观；
- 在陌生环境中自己寻找方向的能力。

所以，“100项未来技能”这个思路是正确的，但要设计成一个**分层能力本体**，而不是平铺的一张清单。

- 自主开始；

# 三、每一项未来技能必须继续向下拆

仅有100个名称还不够。每项技能都应该有完整的数据结构。

例如：

```
未来技能：复杂问题拆解

定义：
把一个模糊、困难或庞大的问题，拆成几个可以执行的小问题。

未来价值：
在AI、项目工作和不确定环境中，人需要先定义问题，再调动工具解决。

6—8岁表现：
能把“整理房间”分成玩具、书、衣服三个步骤。

9—12岁表现：
能把“做一个Minecraft村庄”拆成地图、材料、建筑、测试。

13—15岁表现：
能把一个学习项目拆成目标、资料、实验、作品、展示。

可训练载体：
整理、做饭、搭建、编程、写故事、研究项目。

可观察证据：
是否主动拆分；
拆分是否完整；
是否能按顺序执行；
遇到问题是否会重新调整。

可生成任务：
“把你想做的事情拆成三张任务卡。”

可形成作品：
任务地图、流程图、项目计划、成品记录。
```

也就是说，每项未来能力要形成：

```
能力定义
→ 年龄发展阶梯
→ 可观察行为
→ 当前可学知识
→ 可训练活动
→ 真实任务
→ 作品证据
→ 评估规则
```

这才是系统真正的知识壁垒。

# 四、你说的“天赋驱动系统”应该怎样设计

你的天赋系统不应只是一次问卷，也不应该让AI问几道题后直接宣布：

> “你的孩子有某种天赋。”

更科学的方式是建立一个**动态成长画像和天赋假设系统**。

## 画像至少要分开六种信息

### 1. 兴趣

孩子愿意主动接近什么：

- 故事；
- 游戏；
- 机械；
- 数字；
- 绘画；
- 人际；
- 运动；
- 自然。

### 2. 潜在优势

孩子在哪些活动中：

- 理解更快；
- 更容易发现规律；
- 能坚持更久；
- 能做出更好的成果；
- 可以迁移到新场景。

### 3. 动力来源

孩子为什么愿意继续：

- 自己选择；
- 好奇结果；
- 看见进步；
- 获得认可；
- 做出作品；
- 与人合作；
- 解决真实问题。

### 4. 当前卡点

例如：

- 不愿开始；
- 遇到困难就停；
- 容易分心；
- 怕做不好；
- 会做但不会表达；
- 兴趣多但没有成果；
- 执行快但缺少思考；
- 依赖成人提醒。

### 5. 支持方式

不是把孩子定义成某种“学习风格”，而是记录：

- 什么时候适合提示；
- 什么时候应该让他自己试；
- 任务要拆多小；
- 文字、示范还是实物更有效；
- 独立完成还是伙伴协作更有效。

### 6. 家庭与现实条件

例如：

- 每天可用时间；
- 家长陪伴能力；
- 设备条件；
- 书籍和材料；
- 学校压力；
- 身体情况；
- 居住环境；
- 可获得的老师和同伴。

# 五、画像不能只靠“说”，还要靠“做”

你的系统可以从四类证据建立画像。

## A. 主观描述

- 家长描述；
- 孩子自述；
- AI逐步提问；
- 孩子喜欢和不喜欢什么。

## B. 行为数据

- 哪类任务主动选择；
- 哪类任务经常换掉；
- 在什么任务上坚持更久；
- 哪类任务需要更多提示；
- 哪些任务经常主动做第二版。

## C. 作品数据

- 写的故事；
- 搭建作品；
- 录音讲解；
- 画作；
- 编程程序；
- 观察报告；
- 项目成果。

## D. 复盘数据

- 难不难；
- 有没有趣；
- 哪里卡住；
- 用了什么方法；
- 下次愿不愿意继续；
- 能不能迁移到别的问题。

由此，系统输出的不能是：

> “孩子是创造型天才。”

而应是：

> “目前有中等证据表明，孩子在有明确世界观和可见成果的创造活动中投入度较高；仍需通过不同材料和不同场景继续验证。”

这种表达既保留了“发现天赋”的价值，又避免过早定型。

# 六、未来技能与孩子画像怎样结合

这一步是你的核心算法。

可以理解为两个知识图谱的匹配：

## 图谱一：未来能力图谱

```
未来能力
→ 子能力
→ 年龄表现
→ 可训练活动
→ 所需知识
→ 可形成作品
```

## 图谱二：儿童成长图谱

```
兴趣
→ 优势假设
→ 当前问题
→ 动力来源
→ 支持方式
→ 环境资源
→ 已有作品和行为证据
```

规划系统在两者之间寻找最佳路径：

```
孩子愿意做什么
+
孩子当前最需要解决什么
+
未来哪些能力最重要
+
家庭现实中能做到什么
=
当前最值得发展的成长路线
```

这比单纯补短板更合理。

系统不能只问：

> 孩子哪里不行？

还要问：

> 能不能利用孩子已经喜欢的东西，顺便训练他未来最需要的能力？

# 七、举一个具体例子

假设一个9岁孩子：

- 喜欢Minecraft；
- 喜欢故事和角色；
- 阅读量较大；
- 遇到难题容易停住；
- 容易忘记收尾；
- 不喜欢纯练习；
- 做出作品后愿意向别人展示。

系统不应简单输出：

> 多练数学、多写作文、少玩游戏。

它应该识别：

## 天赋和兴趣入口

- 世界观和故事；
- 空间搭建；
- 规则系统；
- 深度兴趣；
- 作品展示。

## 当前发展问题

- 任务拆解弱；
- 遇难切换策略弱；
- 收尾和检查弱；
- 兴趣尚未稳定转化为成果。

## 匹配的未来能力

- 系统思维；
- 项目管理；
- 空间推理；
- 资源预算；
- 复杂问题拆解；
- 迭代；
- 表达与展示；
- AI协作；
- 自我调节。

## 形成成长蓝图

主项目可以不是“学习任务”，而是：

> 用三个月设计、建造并展示一座有规则、有资源系统、有故事背景的Minecraft城市。

这个项目自然包含：

- 数学：面积、比例、材料预算；
- 写作：城市背景和人物故事；
- 表达：讲解设计；
- 编程：命令、红石或模组；
- 数据：统计材料和时间；
- 项目能力：计划、版本、检查；
- AI能力：让AI提供方案，但自己比较和验证；
- 协作：邀请家人或同伴参与；
- 复盘：哪里失败，下一版怎么改。

这就是你一直强调的：

> **熟悉外壳＋真实目标＋完整工作流＋综合能力嵌入。**

系统不是命令孩子“训练系统思维”，而是通过一个孩子真正愿意完成的项目，使系统思维自然出现。

# 八、蓝图应该怎样逐层拆解

你的层级思路正确，但不能把20年、年、月、周、日全部一次性写死。

## 1. 10—20年：能力北极星

不是预测孩子将来当什么职业，而是建立能力组合：

```
能自主学习
能与AI协作
能定义和解决问题
能创造作品
能与人合作
能管理身心
能做负责任的判断
```

未来本身不可精确预测，OECD也明确强调，教育的重点不是给孩子固定指令，而是让他们能够在陌生情境中自己导航。

所以这里应叫：

> **未来能力罗盘**

而不是20年职业规划。

## 2. 3—5年：发展阶段蓝图

例如9—12岁阶段：

- 建立自主学习；
- 形成2—3个深度兴趣；
- 完成若干真实项目；
- 掌握基础数字和AI工具；
- 发展稳定运动习惯；
- 学会表达和展示成果。

## 3. 年计划：年度成长主题

一年不要发展100项技能。

建议只选择：

- 2项底座能力；
- 2项探索能力；
- 1项身心能力；
- 2—4个年度项目。

例如：

```
底座：
任务闭环、困难切换

探索：
系统设计、表达展示

身心：
姿态与核心力量

年度项目：
Minecraft城市
家庭旅行规划
个人故事集
AI真假侦探实验册
```

## 4. 月计划：阶段性项目

每个月围绕一个真实成果：

- 作品；
- 表演；
- 展览；
- 程序；
- 调查报告；
- 家庭服务；
- 比赛；
- 一次真实解决问题。

## 5. 周计划：项目里程碑

一周只回答：

- 本周推进哪个项目；
- 完成哪个看得见的阶段；
- 重点练哪一种方法；
- 留下什么证据。

## 6. 日计划：行动执行

日计划进入你说的：

- Todo List；
- 番茄钟；
- 任务拆解；
- 时间安排；
- 完成确认；
- 经验和宝石；
- 卡住救援；
- 简短复盘。

所以：

> 年、月、周、日不是四套独立计划，而是同一条成长路线的不同缩放级别。

------

# 九、番茄钟和Todo List应该处在什么位置

它们不应该成为产品核心，而是蓝图落地的执行器。

每一个日任务必须能向上追溯：

```
今天任务
→ 本周里程碑
→ 本月项目
→ 年度主题
→ 当前成长蓝图
→ 未来能力
```

例如：

```
今日任务：
画出Minecraft城墙第一版

训练能力：
空间规划、任务拆解

本周里程碑：
完成城市防御系统草图

本月项目：
建造城市第一版

年度主题：
把兴趣转化为可展示项目

未来能力：
系统思维、创造项目、资源管理
```

孩子看到的是：

> 今天画一个图。

家长看到的是：

> 这个任务正在训练系统思维和项目拆解。

系统知道的是：

> 这次行为将形成新的成长证据。

这三种界面必须分开。

# 十、游戏化激励系统怎样设计

你的经验、宝石和礼物设计可以保留，而且它对儿童持续执行很重要。

但要建立三层奖励。

## 第一层：即时反馈

完成任务获得：

- 经验；
- 宝石；
- 动画；
- 进度变化；
- 技能升级提示。

## 第二层：成果反馈

完成项目获得：

- 作品卡；
- 成长徽章；
- 项目地图；
- 版本轨迹；
- 展示机会；
- 家庭庆祝。

## 第三层：现实奖励

由家长设置兑换：

- 游戏时间；
- 零花钱；
- 小礼物；
- 外出活动；
- 自主选择家庭项目；
- 购买项目材料。

奖励不应只看“完成数量”，还应该奖励：

- 主动开始；
- 遇到困难后继续；
- 自己检查；
- 做第二版；
- 留下作品；
- 真实复盘；
- 把方法迁移到新场景；
- 帮助别人。

不建议用高频随机宝箱作为主要动力。可以有惊喜，但更适合绑定在真实里程碑，而不是每次点击后的随机概率。

# 十一、我建议把整个系统设计成六个核心引擎

## 1. 未来能力引擎

负责：

- 维护100项未来能力；
- 引用研究、产业和教育框架；
- 每年或每半年更新版本；
- 标注能力的稳定性和证据强度；
- 建立年龄发展阶梯。

不能让大模型每次临时“想出100项能力”，否则不同时间会得到不同答案。

应形成版本化知识库，例如：

```
Future Skills Map 2026.1
Future Skills Map 2027.1
```

## 2. 天赋画像引擎

负责：

- AI提问；
- 家长输入；
- 孩子自述；
- 行为观察；
- 作品分析；
- 问题识别；
- 优势和天赋假设；
- 支持方式；
- 置信度和反证。

## 3. 蓝图规划引擎

负责：

- 把未来能力与孩子画像匹配；
- 选择当前重点；
- 形成长期能力罗盘；
- 生成阶段蓝图；
- 制定年度成长主题；
- 选择适合孩子的项目路径。

## 4. 分层计划引擎

负责：

```
年计划
→ 季度/月度项目
→ 周里程碑
→ 日行动
```

计划应滚动调整，而不是一次生成后永久执行。

## 5. 行动与激励引擎

负责：

- Todo；
- 番茄钟；
- 专注；
- 提醒；
- 宝石；
- 经验；
- 等级；
- 礼物兑换；
- 项目进度；
- 任务协商。

## 6. 证据与进化引擎

负责：

- 记录作品；
- 记录完成行为；
- 记录复盘；
- 验证原来的天赋假设；
- 发现新的优势；
- 识别计划不适合；
- 动态修改画像；
- 重算蓝图和后续计划。

完整闭环是：

```
未来能力库
        ↓
儿童初始画像
        ↓
成长蓝图
        ↓
年、月、周、日计划
        ↓
真实任务与项目
        ↓
作品、行为、复盘证据
        ↓
更新画像
        ↓
调整成长蓝图
```

# 十二、对当前产品最关键的改造建议

## 1. 不要删除画像优先，而是区分“家长建档”和“孩子使用”

你坚持先画像是有道理的。

但可以分成两阶段：

### 家长建档阶段

家长完成：

- 基本情况；
- 兴趣和优势；
- 当前问题；
- 身体和环境限制；
- 对孩子的观察；
- 期望和担忧。

### 孩子认识阶段

孩子以对话、小游戏、小任务完成：

- 自我描述；
- 兴趣选择；
- 动力测试；
- 困难反应；
- 作品尝试；
- 对AI画像的纠正。

这样既保留深度画像，又不会让一个6岁孩子连续完成成人式问卷。

## 2. 当前8项能力可以保留为一级展示，但后台扩展到100项

现在的8项能力太粗，不能支持真正个性化规划。

但不应该在孩子端直接显示100项。

建议：

```
后台：100项未来能力
家长端：10个能力域＋当前重点
孩子端：3—5颗正在升级的技能星
```

## 3. 把当前所有功能重新定义为“证据入口”

现有的：

- 日记；
- 灵感；
- 作品；
- 任务；
- 番茄钟；
- 复盘；
- 习惯；
- AI问答。

不要彼此争夺主导航。

它们本质上都是：

> 帮助画像和蓝图获得真实证据的不同入口。

## 4. 建立“计划为什么存在”的可解释链路

每个计划都必须显示：

- 来自哪条画像证据；
- 要发展哪项未来能力；
- 为什么选择这个项目；
- 什么情况会调整；
- 孩子是否同意；
- 计划成功的可观察标准。

## 5. 允许孩子修改蓝图

未来蓝图不能成为AI和家长共同制定、孩子只能服从的东西。

孩子应该可以说：

- 这个不像我；
- 我不想走这条；
- 我最近喜欢上了别的；
- 这个目标太难；
- 我想换一种作品；
- 我想暂停。

这与OECD强调的学生自主性是一致的：未来教育不只是替学生规定方向，而是帮助他们逐渐学会自己导航。

# 十三、这套产品最大的三个风险

## 风险一：100项未来能力变成另一个应试体系

如果家长看到100项，就可能产生：

> 我的孩子这项只有40分，必须补齐。

最终会变成比学校课程更复杂的考核表。

因此必须强调：

- 不要求全部发展；
- 不追求平均；
- 允许形成独特能力组合；
- 关注迁移和真实成果；
- 重点是找到最适合孩子的成长路径。

## 风险二：AI过早给孩子贴“天赋”标签

“天赋”很有吸引力，但也是危险词。

建议产品语言采用：

- 天赋线索；
- 优势假设；
- 潜在发展方向；
- 需要继续验证；
- 当前高投入场景。

而不是：

- 你就是某种人；
- 你不适合某件事；
- 你未来应该从事某职业。

## 风险三：规划非常宏大，实际执行失去关联

如果系统生成了漂亮的20年蓝图、年计划、月计划，但孩子每天只是在勾选与蓝图无关的任务，整个系统会退化成包装复杂的Todo。

所以必须建立硬规则：

> 任何任务都必须能够向上追溯到项目、计划和未来能力；任何未来能力都必须能够向下落到真实任务和作品。

# 最终产品定位

我认为你真正应该做的不是“成长OS”这么泛的东西，而是：

> **一套以未来能力为方向、以儿童天赋和问题为起点、以真实项目为载体、以分层计划为路径、以游戏化执行为动力、以长期证据不断自我修正的AI儿童成长操作系统。**

它最重要的不是“AI每天给孩子布置任务”，而是完成这条链：

```
未来需要什么
→ 我的孩子是谁
→ 他适合怎样成长
→ 未来几年走哪条路线
→ 今年做哪些项目
→ 这个月完成什么成果
→ 这周推进哪一步
→ 今天具体做什么
→ 做完后证明了什么
→ 下一步怎样变得更适合
```

## 最值得沉淀的设计原则

**关键假设：** 未来能力可以抽象，但不能精确预测，因此系统应维护“可更新的能力地图”，而不是宣称拥有唯一正确答案。

**核心壁垒：** 不是100项技能名称，而是“未来能力—年龄表现—学习载体—真实任务—作品证据—画像更新”之间的知识图谱。

**产品盲区：** 家长可能把成长蓝图理解成另一套成绩排名，所以必须把“独特组合、持续验证、孩子自主选择”写入产品机制。

**最低成本验证：** 先选10个未来能力，为一个孩子建立画像，生成一个月蓝图，验证能否稳定地从蓝图生成周计划、日任务，并让完成证据反向修正画像。

**决策规则：**

> 任何功能都必须至少服务于“认识孩子、选择未来能力、形成蓝图、执行计划、产生证据、修正画像”中的一个环节；否则不进入主系统。

成长OS不是一个单纯的儿童待办清单，也不是让孩子为了奖励机械打卡的抽奖游戏。

它是一套：

> **以未来100项能力为方向，以儿童动态画像为起点，以真实项目与任务为训练载体，以每日小Boss和每周Boss保持动力，以作品和行为证据不断修正成长蓝图的AI儿童成长操作系统。**

完整闭环：

```text
未来能力地图
→ 儿童画像与天赋线索
→ 个性化成长蓝图
→ 年计划 / 月项目 / 周目标 / 日行动
→ 完成核心打卡
→ 每日小Boss
→ 每周能力Boss
→ 作品、复盘与迁移证据
→ 更新画像与蓝图
```

本设计文件重点定义最底层的“执行与动力系统”：

- 10个成长世界；
- 100个能力Boss；
- 每日小Boss解锁规则；
- 每周Boss战节奏；
- Boss血量与证据攻击；
- 经验、宝石、道具和100种强奖励；
- 家长审批、安全边界和程序数据结构。


## 1. 总体游戏结构

### 1.1 100个Boss不是100个连续周

系统拥有100个能力Boss，分别对应100项未来能力。AI与规则引擎根据孩子当前蓝图，从Boss库中选择本周最适合挑战的一只。

Boss可以重复出现，但每次难度不同：

| 难度 | 含义 | 主要要求 |
|---|---|---|
| 青铜 | 第一次认识并模仿能力 | 能在提示下完成 |
| 白银 | 独立使用能力 | 减少成人帮助 |
| 黄金 | 换场景迁移 | 在第二种场景使用 |
| 钻石 | 综合项目应用 | 与其他能力组合 |
| 传说 | 主动教会别人或创造新方法 | 产生高质量作品与迁移证据 |

因此，系统不是“100周毕业”，而是围绕孩子成长反复选择、升级和验证。

### 1.2 每周一个大Boss

每周只设定一个主Boss。这个Boss对应本周重点能力，并向上关联：

```text
本周Boss
→ 本月项目
→ 年度成长主题
→ 当前成长蓝图
→ 未来能力地图
```

### 1.3 每天一个小Boss

周一至周六，孩子完成当日全部“核心打卡”后，解锁一个2—8分钟的小Boss。

小Boss不是额外增加一份作业，而是：

- 对当天所练能力进行趣味检验；
- 让机械打卡变成剧情高潮；
- 获得攻击周Boss的符文；
- 形成一条轻量成长证据；
- 获得少量即时奖励。

周日不再安排普通小Boss，而是进行本周大Boss决战。

### 1.4 核心打卡与支线任务

每天任务分为两层。

**核心打卡：**

- 每天1—3项；
- 由蓝图、学校/家庭责任和身体状态共同决定；
- 全部完成或经合理协商调整后完成，才能解锁小Boss。

**自选支线：**

- 阅读、运动、帮助家人、项目创作、探索等；
- 不影响小Boss资格；
- 提供额外经验、宝石和装饰材料；
- 不得用大量支线替代核心责任。

“全部打卡”指完成当日经过确认的合理计划，不是必须僵硬完成原始任务。生病、临时事件、错误任务和低能量状态可以通过缩小、替换、延期或恢复模式调整。


## 2. 每周Boss标准节奏

| 日期 | 剧情阶段 | 核心目标 | 解锁内容 |
|---|---|---|---|
| 周一 | 发现Boss | 认识本周能力和Boss弱点 | 弱点卡 |
| 周二 | 学习武器 | 学习或模仿一种方法 | 方法符文 |
| 周三 | 第一次交锋 | 独立使用一次 | 行动符文 |
| 周四 | Boss反击 | 处理一个真实困难 | 韧性符文 |
| 周五 | 换场景追击 | 在第二种场景使用 | 迁移符文 |
| 周六 | 锻造终极武器 | 留下作品、记录或可观察成果 | 成果武器 |
| 周日 | 大Boss决战 | 综合挑战＋孩子复盘 | 强奖励宝箱 |

允许家庭根据作息调整日期，但必须保留“认识—练习—独立—困难—迁移—成果—决战”的逻辑。

### 2.1 周Boss血量

标准周Boss拥有：

- 6层护盾；
- 40点核心生命值。

周一至周六，每击败一只每日小Boss：

- 打破1层护盾；
- 获得1枚证据符文；
- 对核心造成0—5点额外伤害。

周日决战主要依据真实证据造成伤害：

| 证据攻击 | 最高伤害 | 判断依据 |
|---|---:|---|
| 启动之剑 | 8 | 是否自主开始 |
| 方法法杖 | 8 | 是否选择并使用有效方法 |
| 韧性之盾 | 8 | 遇到困难后是否调整继续 |
| 迁移之弓 | 8 | 是否在第二场景使用 |
| 成果水晶 | 12 | 是否留下真实作品或可观察成果 |
| 复盘之眼 | 6 | 是否知道自己用了什么、下次怎样调整 |

不要求每周满分。达到40点即击败Boss；达到50点以上为“完美胜利”。

如果证据不足，Boss不会宣布孩子失败，而是进入：

- 撤退：下周换更小Boss；
- 休眠：保留进度稍后继续；
- 变形：改为更适合孩子兴趣的挑战；
- 降阶：从黄金降为白银难度。

### 2.2 五种证据等级

| 等级 | 证据类型 | 示例 |
|---|---|---|
| E1 | 口头或选择 | 孩子能说出方法 |
| E2 | 行为记录 | 真实完成一次 |
| E3 | 独立行为 | 无提醒或少提示完成 |
| E4 | 迁移证据 | 在另一场景再次使用 |
| E5 | 作品与教学 | 留下成果或教会别人 |

Boss难度越高，需要的证据等级越高。


## 3. 每日小Boss系统

### 3.1 解锁条件

满足以下任一种情况即可解锁：

1. 当日全部核心任务完成；
2. 当日计划经过孩子和家长协商缩小或替换，并完成调整后的计划；
3. 处于生病或恢复模式，完成系统设定的恢复核心任务；
4. 家长确认原任务不合理并撤回，剩余核心计划完成。

以下情况不能解锁：

- 只点击完成但没有任何可观察行为；
- 用大量简单支线刷经验替代核心任务；
- 家长代做；
- AI直接生成成品，孩子没有参与；
- 通过付费或消耗宝石跳过全部核心责任。

### 3.2 小Boss七种模板

1. **闪电回答型**：回答一个与当日方法有关的问题。
2. **限时行动型**：2—5分钟完成一个现实小动作。
3. **策略选择型**：面对困难从多个方法中作出选择并说明理由。
4. **真假侦探型**：判断事实、观点、AI幻觉或错误策略。
5. **微型创作型**：画、写、录或搭建一个极小成果。
6. **身体反击型**：完成适龄动作、呼吸、平衡或恢复挑战。
7. **家庭协作型**：向家人讲解、感谢、求助或共同完成最后一击。

### 3.3 小Boss奖励

标准掉落：

- 5—12经验；
- 1—2宝石；
- 1枚周Boss符文；
- 小概率掉落装饰碎片、故事碎片或技能卡。

小Boss不掉落“免写作业卡、免洗碗券”等强奖励。强奖励只来自每周大Boss，避免每天强刺激。


## 4. Boss选取算法

每周Boss由确定性规则评分，AI只能在候选Boss中负责故事化表达，不得自由创建与蓝图无关的Boss。

建议评分：

```text
Boss适配分 =
成长蓝图优先级 × 30%
+ 当前问题强度 × 20%
+ 孩子兴趣匹配 × 15%
+ 上周证据缺口 × 15%
+ 本月项目关联 × 10%
+ 年龄适配 × 5%
+ 家庭资源可执行性 × 5%
```

限制规则：

- 同一个Boss最多连续出现2周；
- 同一能力域连续最多3周；
- 身心状态异常时优先降低强度；
- 新画像证据不能立刻推翻长期蓝图；
- 一次失败不得被解释为缺乏天赋；
- 孩子可以拒绝Boss并说明“太难、没兴趣、现在不合适、第一步不清楚”；
- 连续两次出现相同拒绝原因，才调整后续推荐。


## 5. 10个世界与100个能力Boss


### 第一世界：行动荒原

**能力域：自我管理与行动力**  
这里的敌人不会阻止孩子知道答案，而是让孩子“知道却迟迟不开始、开始却不能收尾”。


#### B001｜起步史莱姆

- **对应未来能力：** 自主开始

- **Boss等级：** 标准周Boss

- **Boss攻击：** 不断说“等一会儿”，让任务永远停在零。

- **本周胜利证据：** 在两分钟内自主开始至少3次

- **周日决战：** 孩子选一个真实任务，在没有提醒的情况下说出第一步并立刻行动2分钟。

- **偏向掉落：** 启动类奖励、对应技能徽章、经验与宝石


#### B002｜目标迷雾兽

- **对应未来能力：** 目标意识

- **Boss等级：** 标准周Boss

- **Boss攻击：** 让孩子只看到“要做很多”，看不到今天到底要完成什么。

- **本周胜利证据：** 把模糊要求改写成一个可观察的完成标准

## 6. Boss战剧情生成规则

AI可以为Boss生成：

- 本周开场白；
- 每日小Boss台词；
- Boss弱点提示；
- 成功动画文案；
- 失败后的缩小建议；
- 与孩子兴趣相关的世界皮肤。

AI不得生成：

- 羞辱、恐吓、人格标签；
- “你很懒、你不聪明、你没有天赋”等判断；
- 让孩子隐瞒家长、老师或安全问题的任务；
- 高风险身体动作；
- 上传敏感个人信息的任务；
- 以不完成就失去亲情、尊重或基本权利为威胁；
- 擅自承诺家长必须兑现的金钱或强奖励。

Boss是孩子面对的外部困难，不是孩子本人。

正确表达：

> 拖延巨兽正在用“等一会儿”攻击你。我们只需要挥出第一步。

错误表达：

> 你就是一个拖延的人。


## 7. 奖励系统总体设计

### 7.1 三层奖励

**即时奖励：**

- 经验；
- 宝石；
- 动画；
- 符文；
- 小型装饰。

**成长奖励：**

- 能力徽章；
- Boss战纪念卡；
- 作品收藏；
- 第一版与第二版轨迹；
- 新世界、皮肤和故事章节。

**现实强奖励：**

- 仅在每周Boss击败后出现；
- 从家长提前批准的奖励池中随机抽取三项；
- 孩子三选一；
- 高价值奖励需要家长二次确认。

### 7.2 掉落稀有度

| 稀有度 | 概率建议 | 示例 |
|---|---:|---|
| 精良 | 50% | 菜单选择、贴纸、加15分钟兴趣时间 |
| 稀有 | 30% | 免洗碗券、电影导演卡、材料采购 |
| 史诗 | 15% | 游戏加30分钟、家庭活动设计、作品装框 |
| 传说 | 5% | 半日活动设计、传说级愿望券 |

保护机制：

- 连续3周没有史诗选项，第4周至少出现1项史诗；
- 传说奖励每月最多出现1次；
- 不允许付费购买重抽；
- 概率与奖励池向家长公开；
- 未批准奖励永远不参与抽取；
- 抽中后不自动生效，进入“待预约”状态。

## 9. 奖励券规则

每张现实奖励券包含状态：

```text
未获得
→ 已获得
→ 待预约
→ 家长已确认
→ 已兑现
→ 已过期 / 已转赠
```

必要字段：

```json
{
  "id": "reward_021",
  "name": "免洗碗铁券",
  "category": "housework",
  "rarity": "rare",
  "enabled": true,
  "guardianApprovalRequired": true,
  "cooldownDays": 14,
  "monthlyLimit": 1,
  "estimatedMoneyCost": 0,
  "estimatedTimeCostMinutes": 15,
  "validDays": 30,
  "ageMin": 6,
  "ageMax": 15,
  "cannotUseOn": [],
  "transferable": true
}
```

家长可以配置：

- 开启或关闭；
- 月度限量；
- 金钱预算；
- 时间预算；
- 是否允许转赠；
- 是否允许合成；
- 是否需要预约；
- 哪些日期不能使用；
- 兄妹是否共用奖励池。

### 9.1 奖励边界

不能成为奖励的基本权利：

- 不被辱骂、羞辱或暴力对待；
- 基本陪伴、照顾和安全；
- 必要睡眠、饮食、医疗和教育支持；
- 合理表达和解释机会。

因此“免挨骂金牌”的程序效果必须是：

> 启动冷静沟通流程、暂停训话、双方重新表达。

它不是“孩子表现好才有资格不挨骂”。

“免写作业卡”只能免除家长额外加练，不能擅自免除学校必须完成的作业。产品内正式名称建议为“免家庭加练卡”。


## 10. 经验、等级与宝石经济

### 10.1 经验来源建议

| 行为 | 经验 |
|---|---:|
| 完成普通核心行动 | 8—25 |
| 完成每日小Boss | 5—12 |
| 主动开始 | 额外3 |
| 处理困难并继续 | 额外5 |
| 留下真实作品 | 8—20 |
| 完成复盘 | 3—8 |
| 换场景迁移 | 10—20 |
| 击败周Boss | 40—100 |
| 完美胜利 | 额外20 |

### 10.2 防刷规则

- 同一低难度任务每日只计一次；
- 自建任务每日经验设上限；
- 删除或撤销完成同步撤回奖励；
- 专注时间本身不直接换经验；
- 作品奖励要经过存在性检查；
- AI生成整篇作品而孩子未参与，不计作品经验；
- 家长可以标记异常奖励并回滚。

### 10.3 宝石用途

宝石用于数字世界内部：

- 解锁主题；
- 购买角色服装；
- 解锁Boss图鉴；
- 购买故事章节；
- 装饰作品馆；
- 兑换普通数字道具。

宝石不能直接无限兑换现实金钱或高价值奖励。现实奖励主要来自每周Boss强奖励宝箱。

孩子端只显示：

## 15. AI与程序职责边界

### AI负责

- 在固定100Boss库中推荐候选；
- 根据孩子兴趣生成Boss皮肤和台词；
- 在模板范围内生成每日小Boss题目；
- 将孩子复盘整理成暂时线索；
- 信息不足时追问最多一个问题；
- 根据失败原因提出缩小任务建议。

### 程序负责

- Boss选择评分；
- 解锁条件；
- 经验和宝石结算；
- 护盾与血量；
- 奖励概率；
- 家长审批；
- 数据权限；
- 完成、撤销和删除；
- 证据关联；
- 冷却和月度限量；
- 隐私隔离。

模型不得决定孩子是否真的完成任务，也不得自行授予现实奖励。


## 16. 安全与心理学规则

1. Boss代表外部困难，不代表孩子人格。
2. 不使用“懒、笨、没用、胆小、坏孩子”等标签。
3. 不因休息、延期或合理缩小任务扣经验。
4. 不制造连续签到中断后的损失恐惧。
5. 不将父母的爱、尊重、基本陪伴和不受辱骂变成有条件奖励。
6. 不将100项能力做成百分制排名和补短板清单。
7. 每个阶段最多重点显示3—5项能力。
8. 天赋只能称为“线索、假设或高投入场景”，必须允许反证。
9. 身体挑战必须按年龄和健康条件过滤。
10. 私人日记默认不作为Boss战强制证据。
11. 照片、录音和文本的AI权限必须逐项可关闭。
12. 删除原始证据后，相关画像推断和Boss能力结论必须同步降级或撤回。

```text
→ 奖励进入背包并最终兑现
```

### 必测边界

- 当日任务未全部完成，小Boss保持锁定；
- 任务合理缩小后完成，小Boss可解锁；
- 任务被家长认定不合理并撤回，不惩罚孩子；
- AI不可用时，本地模板仍可生成小Boss；
- 重复点击不重复奖励；
- 撤销完成同步撤回符文；
- 周Boss未被击败时不能抽强奖励；
- 禁用奖励不参与抽取；
- 史诗保底规则正确；
- 传说奖励月度上限正确；
- 兄妹账户和奖励严格隔离；
- 私人日记不进入Boss证据；
- 删除作品后相关证据和技能进度重算；
- 移动端无横向溢出；
- 模态框、奖励三选一和家长确认可键盘操作。


## 19. 产品成功指标

不只统计“打卡次数”，重点观察：

- 核心计划完成率；
- 孩子主动开始率；
- 小Boss解锁率；
- 周Boss参与率；
- 周Boss击败率；
- 任务被缩小或替换的原因；
- 真实作品数量；
- 迁移证据数量；
- 孩子主动使用方法的次数；
- 家长催促时间变化；
- 奖励兑现率；
- 现实奖励是否逐渐减少而内在和成果动力上升。

产品最终目标不是让孩子永远依赖奖励，而是让奖励逐渐从：

```text
我要得到卡券
→ 我想打败Boss
→ 我想完成作品
→ 我知道自己正在变强
→ 我可以自己选择目标和方法
```

## 20. 最终设计原则

> 普通任务训练行动，每日小Boss制造乐趣，每周大Boss验证能力，随机强奖励制造期待，真实作品和迁移证据决定成长。

任何新增玩法必须至少服务于以下一项：

- 让孩子更愿意自主开始；
- 让未来能力落到真实行动；
- 让孩子在困难后调整继续；
- 产生作品和迁移证据；
- 帮助AI更准确地修正画像；
- 减少家长反复催促；
- 增强家庭中的积极互动。

不满足这些条件的玩法，不进入成长OS主系统。
