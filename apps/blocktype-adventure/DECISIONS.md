# DECISIONS

## D-001 暂存为独立子项目

- 决策：先放入 `growth-os/apps/blocktype-adventure`，使用独立分支开发。
- 原因：当前 GitHub 连接器没有创建新仓库能力，但具备现有仓库分支与文件写入权限。
- 放弃方案：覆盖 growth-os 根项目；等待人工创建空仓库后再开始。
- 重新评估：独立仓库可用后，将该目录完整迁移并保留提交历史或以首个稳定版本导入。

## D-002 第一版使用 Phaser 3

- 决策：使用 Vite + TypeScript + Phaser 3。
- 原因：适合浏览器 2D 游戏，具备场景、输入、时间和动画能力，维护成本低于自研 Canvas 引擎或 Three.js。
- 放弃方案：React/Vue UI 驱动游戏、Three.js 3D 沙盒。
- 重新评估：只有在核心玩法验证通过且 3D 明确提升留存时，才评估 3D。

## D-003 正式素材不得从概念图切片

- 决策：概念图只作视觉参考，正式运行时素材需重新绘制为原创透明 SVG 或程序图形。
- 原因：避免模糊、缩放失真、不可维护及版权风格混淆。
- 放弃方案：从 AI 概念图裁切按钮、角色、怪物和地形。
- 重新评估：无。

## D-004 运行时代码保持 ES2022 基线

- 决策：在项目 `target` 与 `lib` 仍为 ES2022 时，不使用 `toSorted()` 等 ES2023 API；优先采用 ES2022 等价实现。
- 原因：避免类型检查通过条件与目标浏览器运行能力不一致，并保持 Chrome、Edge 的部署兼容性可预测。
- 放弃方案：仅为使用少量新数组 API 将整个项目升级到 ES2023，或引入 polyfill 依赖。
- 重新评估：当浏览器支持范围、Vite 构建目标和 TypeScript `lib` 统一升级时，再整体调整语言基线。

## D-005 输入规则保持单一事实来源

- 决策：目标选择、锁定保持、字符判定、完成解锁、Backspace 和目标移除统一由纯 TypeScript `TypingSystem` 决定；`GameScene` 只负责将敌人映射为输入目标并呈现结果。
- 原因：避免场景代码与单元测试各自维护一套规则，出现“测试通过但实际游戏行为不同”的假绿灯。
- 放弃方案：继续在 `GameScene` 中复制相同判断；直接用 Phaser 对象编写难以隔离的输入测试。
- 重新评估：增加中文输入法或多字符输入时扩展 `TypingSystem` 的接口，但仍保持运行时与测试共用同一核心。

## D-006 浏览器测试通过 Phaser 实例观察真实状态

- 决策：在 `globalThis.__BLOCKTYPE_GAME__` 暴露 Phaser 游戏实例，Playwright 读取真实场景状态，不通过隐藏 DOM 或截图像素复制业务判断。
- 原因：Canvas 内文字和对象不适合用普通 DOM 选择器稳定验证；直接观察场景可以确认键盘事件确实进入运行时。
- 放弃方案：截图像素判断暂停和输入；维护一套与游戏状态重复的隐藏 DOM。
- 风险：全局入口增加少量调试表面积，但不包含用户数据或密钥。
- 重新评估：建立正式只读诊断 API 后替换当前入口。

## D-007 浏览器测试夹具只能准备前置数据，不能伪造结果

- 决策：当随机生成无法稳定构造必要场景时，Playwright 可以在动作发生前规范化少量临时场景数据，例如把两个已真实生成和移动的敌人单词改为同首字符；不得直接设置锁定 ID、输入进度、统计、胜负或结算结果来满足断言。
- 原因：同首字符最近目标需要可重复前置条件，而覆盖全局 `Math.random` 实测不能稳定控制 Phaser 的选词工具。规范化两个单词比复制生成器或在生产代码加入测试分支更小、更可回滚。
- 验证边界：敌人位置与移动、键盘事件、`TypingSystem` 选择、Backspace、场景重开和所有断言必须走生产路径。
- 放弃方案：等待随机碰撞同首字符；覆盖全局随机数；直接改锁定或进度；为单个测试引入生产环境测试模式。
- 重新评估：当游戏提供正式关卡种子、依赖注入或测试夹具 API 后，移除对场景对象的直接前置数据修改。

## D-008 CI 失败保留浏览器诊断，成功不保留

- 决策：Playwright 失败时上传 report、trace、截图和视频，保留 1 天；成功时跳过诊断包，只上传已验证 `dist`。
- 原因：浏览器失败日志经常无法显示完整断言上下文，短期诊断包能降低定位成本，同时避免长期占用 Actions 存储。
- 放弃方案：每次运行都上传视频；只依赖截断的工作流文本日志。
- 重新评估：测试规模增大后按失败类型和敏感信息策略调整保留范围。

## D-009 胜负闭环必须走生产事件链

- 决策：测试只把倒计时、基地生命或敌人位置准备到触发边界，并允许调整场景时钟倍率；胜利仍由倒计时回调触发，失败仍由敌人越界触发，重开仍由 Canvas 按钮点击触发。
- 原因：直接修改胜负状态无法证明玩家流程真实可用。
- 放弃方案：直接设置胜负标志、伪造结算层、绕过按钮调用重开。
- 重新评估：当场景拆分为正式状态机后，改由公开状态转换接口验证。

## D-010 高速输入基线同时验证浏览器事件与游戏结果

- 决策：高速输入测试使用 Playwright 真实键盘事件，并同时记录捕获阶段 `keydown` 顺序和游戏运行时统计；不能只看最终进度或只看浏览器事件数量。
- 原因：浏览器收到事件不代表 Phaser 正确处理，最终进度正确也不能排除中间乱序或错误统计。双侧证据才能定位丢键发生在浏览器派发层还是游戏处理层。
- 测试边界：允许冻结场景时钟、移除额外敌人并设置一个长测试单词；锁定、字符判定、错误统计和完成仍由生产键盘监听与 `TypingSystem` 执行。
- 当前基线：CI Chromium 在 10、20、30 字符/秒下均无丢键、乱序或统计偏差。
- 重新评估：加入中文输入法组合事件、移动端软键盘或更高输入速率时，建立独立基线，不把桌面英文键盘结果外推到其他输入路径。

## D-011 场景重启必须显式传递目标状态

- 决策：`scene.restart()` 的调用方必须显式传入 `{ autoStart: true }` 或 `{ autoStart: false }`，不能依赖缺省值表达“重玩”或“返回首页”。
- 原因：实测 Phaser 会在连续重启中保留上次传入的数据；先点击“再来一局”后再点击“返回首页”，无参数重启会继承 `autoStart:true`，造成首页立即进入战斗。
- 放弃方案：假设 `restart()` 无参数会清空旧数据；在 `create()` 中根据界面对象猜测目标状态。
- 验证：Playwright 先走胜利重玩，再走失败返回首页，确认敌人、统计和 `started` 均恢复到首页状态。
- 重新评估：当开始、战斗、暂停和结算拆为独立 Phaser Scene 或正式状态机时，可移除该重启数据约定。

## D-012 生产域名不得用于未经验证的部署探测

- 决策：后续部署实验优先使用 Preview；只有完整构建产物已准备且可验证时才部署到 Production。若 Production 探测失败或覆盖异常，必须先恢复用户可访问入口，再继续开发功能。
- 原因：Vercel 项目保留 Vite 构建配置，仅上传静态 `index.html` 仍会执行 `vite build` 并失败；Production 探测会直接影响正式别名。
- 当前恢复方案：Production Deployment `dpl_CqxqHuScPqHrYM4X4niWWqvPArz6` 仅转发到已验证可玩的旧 Deployment，明确不代表最新分支已上线。
- 放弃方案：将约 1.2 MB 构建 bundle 人工嵌入工具参数；继续用 Production 试错；把 HTTP 200 的转发页描述成新版本发布。
- 下一步：建立 GitHub Integration、Deploy Hook 或 GitHub Actions + Vercel CLI 的自动发布路径，让 CI 通过的同一提交直接成为可追踪 Deployment。
- 重新评估：稳定自动发布和回滚机制建立后，移除临时转发部署。

## D-013 视觉反馈只消费业务结果

- 决策：锁定轮廓、命中脉冲、错误提示和击破碎片只能由已经确定的场景业务结果触发；视觉对象不得参与目标选择、字符判定、统计、胜负或重开。
- 原因：动画帧率、延迟或销毁时机不应改变游戏规则，否则低性能设备会产生不同玩法结果，也会破坏 `TypingSystem` 的单一事实来源。
- 生命周期：临时反馈必须由 Phaser tween 完成回调主动销毁；持久锁定轮廓跟随敌人容器，并由锁定 ID 显隐。
- 素材边界：当前反馈使用原创程序图形与文字，不从概念图切片，不引入外部素材依赖。
- 放弃方案：等待动画完成后才计分或移除敌人；通过身体颜色本身承担全部锁定状态；创建不销毁的粒子对象。
- 重新评估：加入“减少动画”设置后，保留静态锁定与必要文字反馈，跳过短暂 tween，但业务结果保持完全一致。
